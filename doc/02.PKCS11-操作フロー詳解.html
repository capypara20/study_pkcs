    CHAPTER 1 · 乱数生成 · 概要

    03乱数生成 — 最もシンプルな単発操作

    PKCS#11でもっともシンプルな操作。Init/Finalなしの単発関数1本。バッファサイズだけ指定して、HSM内部の真性乱数生成器（TRNG）から取り出す。

    ▸関数シグネチャ

    C / PKCS#11
    CK_RV C_GenerateRandom(
  CK_SESSION_HANDLE hSession,
  CK_BYTE_PTR        pRandomData,
  CK_ULONG           ulRandomLen
);

    引数は3つだけ。セッション・出力バッファ・サイズ。鍵もメカニズムも不要。

    ▸使われる用途

    • Nonce生成 — 一度きりの番号
    • IV / Salt — 暗号化の初期化ベクトル
    • セッション鍵 — 一時的なAES鍵
    • チャレンジ値 — 認証プロトコル
    • UUID — 一意識別子

    いずれも「真にランダムな値」が必要な場面。CPUのrand()では弱い。

    HSM内部のTRNG（True Random Number Generator）

    HSMの中には物理的なエントロピー源（熱雑音、リング発振器、量子ノイズ等）を使った真性乱数生成器がある。これがHSMを使う理由のひとつで、ソフトウェアの疑似乱数（PRNG）とは別物。

    FIPS 140-3対応HSMでは、TRNGの品質要件が SP 800-90A/B/C に基づき厳格に規定されている。

    出典：[1] PKCS#11 Spec v3.2 §5.18[14] NIST SP 800-90A

    CHAPTER 1 · 乱数生成 · 準備

    04準備フェーズ

    乱数生成の準備はセッションさえ開いていれば十分。ログインさえ原則不要（仕様上）。ただし受け取りバッファをアプリ側で確保しておく必要がある。

    ▸ PREPARATION

    必要な前提条件

    条件必須？備考
    セッション必須C_OpenSession 済み
    ログイン原則不要仕様上はPublic状態でOK。ただしベンダーによっては必須
    鍵ハンドル不要乱数に鍵は関係ない
    メカニズム不要選択肢なし
    出力バッファ必須アプリ側で ulRandomLen 分を事前確保
    バッファサイズの実務
    • 16 bytes = AES-128の鍵やCBC IVに十分
    • 32 bytes = AES-256やSHA-256のソルト相当、最も一般的
    • 大きすぎるサイズ はベンダーによって上限あり（数KB〜MB）
    出典：[1] PKCS#11 Spec v3.2 §5.18.2

    CHAPTER 1 · 乱数生成 · 実行

    05実行フェーズ — シーケンス図で内部を覗く

    アプリが C_GenerateRandom を呼んだ瞬間、3つのレイヤー（App / Library / HSM）でこんなやりとりが起きてる。

    シーケンス図：縦軸が時間、横線がメッセージ、太縦線がアクティブ区間
    ライブラリの仕事

    引数検証 → リクエスト梱包（TLS） → HSMへ送信 → 応答受信 → アプリバッファに書込み → 戻り値返却

    HSMの仕事

    TRNGから物理エントロピー取得 → DRBG（NIST SP 800-90A準拠）で平滑化 → 指定バイト数を返却

    出典：[1] PKCS#11 Spec v3.2 §5.18.2[14] NIST SP 800-90A/B/C

    CHAPTER 1 · 乱数生成 · 結果

    06結果フェーズ + コード例

    戻り値は CKR_OK（成功）または各種エラーコード。バッファは引数で渡したポインタに直接書き込まれる。新たにメモリは確保されない（アプリの責任）。

    ▸ RESULT

    戻り値と状態変化

    戻り値意味
    CKR_OK (0x00)成功。pRandomDataに乱数が格納されている
    CKR_SESSION_HANDLE_INVALIDセッションが無効。要再オープン
    CKR_ARGUMENTS_BADバッファがNULL等の引数不正
    CKR_RANDOM_NO_RNGそのTokenがRNG非対応（極めて稀）
    CKR_DEVICE_ERRORHSM側障害。リトライ検討

    ▸C 言語実装

    C
    // 32バイトの乱数を取得
CK_BYTE buf[32];
CK_RV rv = pFuncs->C_GenerateRandom(
    hSession,
    buf,
    sizeof(buf)
);
if (rv != CKR_OK) {
    fprintf(stderr, "random failed: 0x%lx\n", rv);
    return -1;
}
// buf[0..32] が乱数で埋まっている

    ▸Rust 実装（cryptoki クレート）

    Rust
    use cryptoki::session::Session;

// 32バイトの乱数を取得
let mut buf = [0u8; 32];
session.generate_random_slice(&mut buf)?;

// buf[..] に乱数が入る
println!("{:02x?}", buf);
    セッション状態への影響

    乱数生成はセッション状態を変更しない。C_SignInitのような「次の操作を待ってる状態」を作らないので、何度連続で呼んでも問題ない。これも乱数が「軽い」と言われる理由のひとつ。

    出典：[1] PKCS#11 Spec v3.2 §5.18[13] cryptoki crate

    CHAPTER 1 · 乱数生成 · ★ 性能評価

    07性能評価ポイント

    乱数生成は暗号演算を伴わないため、純粋に「ライブラリのオーバーヘッド + ネットワーク往復 + HSM内のTRNG読み出し」の測定になる。

    ★ どこを測るか

    計測対象は C_GenerateRandom の呼び出し前後の時刻差。Init/Finalがないので測定設計が最もシンプル。

    疑似コード
    // ★ 計測区間
t0 = get_time_ns();
rv = pFuncs->C_GenerateRandom(hSession, buf, len);
t1 = get_time_ns();

latency_ns = t1 - t0;

    ▸測定すべき変数

    • バッファサイズ別：16B / 32B / 256B / 1KB / 16KB
    • 並列度：シングル → マルチスレッド
    • 連続呼び出し：定常状態のスループット
    • セッション数：1セッション vs 複数並列

    ▸取るべきメトリクス

    • レイテンシ（平均、p50、p99）
    • スループット（ops/sec、MB/sec）
    • スケール係数（スレッド数とTPSの比）
    注意：HSM側のRNG上限
    • HSMのTRNGスループットには物理的な上限がある（数MB/sec〜）
    • 大量に乱数を取り続けるとエントロピー枯渇を起こすベンダーもある
    • ネットワーク型HSMではTLS往復のレイテンシが支配的になる（バッファサイズが小さい場合）
    期待される結果の目安（ネットワーク型HSM）
    項目典型値（参考）備考
    レイテンシ（32B）0.5〜2 msRTT支配
    レイテンシ（16KB）2〜10 ms転送量増加
    スループット（単一）500〜2000 ops/secシングルスレッド
    スループット（並列）セッション数倍HSM上限まで

    ※ あくまで一般的な目安。実機・ネットワーク環境で大きく変動する

    出典：[1] PKCS#11 Spec v3.2 §5.18[7] AWS CloudHSM PKCS#11 library[14] NIST SP 800-90B (Entropy)

    CHAPTER 2 · 署名 · 概要

    08署名 — 3段ロケットの実例

    PKCS#11でもっとも使われる操作。C_SignInit で準備、C_Sign で実行。秘密鍵はHSM内にあるまま、署名だけが返ってくる。性能評価のメインターゲット。

    ▸関数シグネチャ（ワンショット版）

    C / PKCS#11
    // 準備
CK_RV C_SignInit(
  CK_SESSION_HANDLE hSession,
  CK_MECHANISM_PTR  pMechanism,
  CK_OBJECT_HANDLE  hKey
);

// 実行
CK_RV C_Sign(
  CK_SESSION_HANDLE hSession,
  CK_BYTE_PTR       pData,
  CK_ULONG          ulDataLen,
  CK_BYTE_PTR       pSignature,
  CK_ULONG_PTR      pulSignatureLen
);

    ▸引数の意味

    • hSession — 利用するセッション
    • pMechanism — どの方式で署名するか（RSA-PSS等）
    • hKey — どの秘密鍵で署名するか
    • pData/ulDataLen — 署名対象のデータ
    • pSignature — 署名結果を書き込むバッファ
    • pulSignatureLen — 入出力兼用：バッファサイズ／実際の署名長
    注文書のたとえ

    "鍵 = #47"・"方式 = RSA-PSS"・"対象データ = ファイル内容" の3点セットを毎回 C_SignInit + C_Sign で渡す。秘密鍵はHSM内部で一時的にロードされ、署名演算後すぐに消える（鍵ハンドルだけが永続）。

    出典：[1] PKCS#11 Spec v3.2 §5.13 Signing functions[3] PKCS#11 Usage Guide v3.2

    CHAPTER 2 · 署名 · 準備

    09準備フェーズ — C_SignInit

    C_SignInitは「これからどの鍵で・どの方式で署名するか」をセッションに記憶させる関数。実際の暗号演算はまだ起きてない。

    ▸ PREPARATION

    必要な前提条件

    条件必須？備考
    セッション必須C_OpenSession 済み
    ログイン必須（Private鍵の場合）C_Login(CKU_USER) 完了済み
    鍵ハンドル必須CKA_SIGN=true の秘密鍵を C_FindObjects 等で取得済
    メカニズム指定必須鍵の種類と整合するメカニズム
    セッション状態「アクティブ操作なし」状態他のxxxInitが走ってない

    ▸メカニズム構造（CK_MECHANISM）の中身

    SignInitで何が起きてるか
    • ライブラリ：引数検証、鍵のCKA_SIGN属性チェック、メカニズムが鍵タイプと整合するか確認
    • ライブラリ：セッション状態を「署名待ち：鍵#47、PSS方式」に変更
    • ベンダーによっては：HSM側にもInit通知を送る（事前準備）
    よくあるエラー
    • CKR_KEY_FUNCTION_NOT_PERMITTED — 鍵にCKA_SIGN=false
    • CKR_MECHANISM_INVALID — HSMが対応してないメカニズム
    • CKR_KEY_TYPE_INCONSISTENT — RSAメカニズムにEC鍵を渡した等
    • CKR_OPERATION_ACTIVE — 別のxxxInitがまだ走ってる
    出典：[1] PKCS#11 Spec v3.2 §5.13.1, §6 Mechanism specifications

    CHAPTER 2 · 署名 · 実行

    10実行フェーズ — シーケンス図

    C_SignInit + C_Sign の完全な流れ。HSM内部で秘密鍵を一時ロード → ハッシュ計算 → 秘密鍵演算 → 署名返却。乱数生成と違い、実際の暗号計算が走るので時間がかかる。

    SignInit (準備) + Sign (実行) の完全シーケンス。HSM内部のステップに注目
    HSM内部の③が時間を食う理由

    RSA秘密鍵演算（CRT使用でも）は大きな数のべき乗剰余計算。RSA-2048で約1300ビットの掛け算を数百回。これがHSMの専用クリプトプロセッサで実行される。ECDSAだとずっと軽く、AES-CMACなら超軽い。

    出典：[1] PKCS#11 Spec v3.2 §5.13[3] PKCS#11 Usage Guide v3.2[15] NIST FIPS 186-5 Digital Signature Standard

    CHAPTER 2 · 署名 · 補論

    11マルチパート vs ワンショット

    PKCS#11は2種類の署名スタイルを提供する。「データ全部一気に渡す」のか「少しずつ渡す」のか。性能評価ではどちらかに統一する必要がある。

    A. ワンショット（推奨：性能評価で使う）
    パターンA
    C_SignInit(hSession, &mech, hKey);
C_Sign(hSession, data, dataLen,
       sig, &sigLen);
// 1回で完結
    • データ全体をメモリに保持
    • HSMへの往復が 1回
    • 性能測定に向く（変数が少ない）
    • 典型サイズ：〜数MB
    B. マルチパート
    パターンB
    C_SignInit(hSession, &mech, hKey);
C_SignUpdate(hSession, chunk1, len1);
C_SignUpdate(hSession, chunk2, len2);
C_SignUpdate(hSession, chunk3, len3);
C_SignFinal(hSession, sig, &sigLen);
    • ストリーミング処理
    • HSMへの往復が Update回数+2回
    • 大きなデータ（GB級）で必須
    • 性能評価では避ける（ノイズが多い）

    ▸マルチパートのシーケンス

    性能評価で見落としがちなポイント

    マルチパートはUpdateごとにネットワーク往復が発生するため、ネットワーク型HSMでは想像以上に遅い。1MBのデータを 1KBチャンクで送ると 1024往復。ワンショットなら 1往復で済む。「ストリーミング = 速い」と直感で考えると逆の結果になる。

    出典：[1] PKCS#11 Spec v3.2 §5.13.2-5.13.4

    CHAPTER 2 · 署名 · 結果

    12結果フェーズ + コード例

    署名はバッファに書き込まれる。2-call convention（バッファサイズ問い合わせ→実署名）を理解しておくと、ベンダー間の挙動差に対応しやすい。

    ▸ RESULT

    2-call convention（バッファサイズ事前確認）

    pSignature = NULLで呼ぶと、pulSignatureLenに必要なバッファサイズが返る。これでアプリは適切なサイズのバッファを確保してから2回目の呼び出しをする。

    C
    // 1回目：サイズだけ取得
CK_ULONG sigLen = 0;
pFuncs->C_Sign(hSession, data, dataLen, NULL, &sigLen);
// → sigLen に必要なバッファサイズが入る (RSA-2048なら 256)

// 2回目：実際に署名
CK_BYTE *sig = malloc(sigLen);
pFuncs->C_Sign(hSession, data, dataLen, sig, &sigLen);
// → sig[0..sigLen] に署名データ

    ▸C 完全実装

    C
    // 署名（RSA-PSS）
CK_RSA_PKCS_PSS_PARAMS pss = {
    .hashAlg   = CKM_SHA256,
    .mgf       = CKG_MGF1_SHA256,
    .sLen      = 32
};
CK_MECHANISM mech = {
    .mechanism      = CKM_RSA_PKCS_PSS,
    .pParameter     = &pss,
    .ulParameterLen = sizeof(pss)
};

pFuncs->C_SignInit(hSession, &mech, hPrivKey);

CK_ULONG sigLen;
pFuncs->C_Sign(hSession, data, dataLen,
              NULL, &sigLen);

CK_BYTE *sig = malloc(sigLen);
CK_RV rv = pFuncs->C_Sign(
    hSession, data, dataLen, sig, &sigLen);

    ▸Rust 実装

    Rust
    use cryptoki::mechanism::{
    Mechanism, rsa::{PkcsMgfType, PkcsPssParams}
};

let pss_params = PkcsPssParams {
    hash_alg: Mechanism::Sha256,
    mgf:      PkcsMgfType::MGF1_SHA256,
    s_len:    32.into(),
};
let mech = Mechanism::RsaPkcsPss(pss_params);

let signature = session.sign(
    &mech,
    priv_key_handle,
    data
)?;
// signature: Vec<u8> (256 bytes for RSA-2048)
    セッション状態のリセット

    C_Signが成功（または失敗）すると、セッション状態は「アクティブ操作なし」に戻る。同じ鍵で再度署名するには、もう一度 C_SignInit から始める必要がある。これが性能評価で「Initを毎回呼ぶか」議論のポイント。

    出典：[1] PKCS#11 Spec v3.2 §5.13.2[9] AWS CloudHSM PKCS#11 examples[13] cryptoki crate

    CHAPTER 2 · 署名 · ★ 性能評価

    13性能評価ポイント

    署名は性能評価のメインターゲット。実運用で最も呼ばれる操作で、HSMのスペックの差が一番出る。「鍵長」「メカニズム」「並列度」の3軸で測る。

    ★ どこを測るか — 2つの設計
    Init込み (現実的)
    // 毎回 SignInit + Sign を測る
t0 = now();
C_SignInit(s, &mech, hKey);
C_Sign(s, data, len, sig, &sLen);
t1 = now();
// → 実運用に近い
    Sign単体 (純粋計測)
    // SignInitを事前に1回だけ呼んで
// Sign のみを繰り返し測る
C_SignInit(...);  // 一度だけ
for i in 0..N {
   t0 = now();
   C_Sign(...);
   t1 = now();
}

    ※ 注：Sign後にセッション状態がリセットされるので、純粋計測パターンは通常成立しない。ベンダー実装によっては成立する場合もあるが、原則「Init込み」で測る。

    ▸測定すべき変数（マトリクス）

    • アルゴリズム：RSA / ECDSA / EdDSA / HMAC
    • 鍵長：
      RSA-2048 / 3072 / 4096
      EC P-256 / P-384 / P-521
    • メカニズム：RSA-PKCS / RSA-PSS / SHA256-RSA-PKCS
    • データサイズ：32B / 1KB / 16KB / 1MB
    • 並列度：1 / 4 / 16 / 64 スレッド

    ▸取るべきメトリクス

    • レイテンシ分布：mean / p50 / p95 / p99 / max
    • スループット：ops/sec（全スレッド合計）
    • スレッド数によるスケール曲線
    • エラー率（過負荷時の失敗）
    • HSM側CPU使用率（取れれば）
    期待される結果の目安
    アルゴリズム典型レイテンシ典型TPS（単一）備考
    RSA-2048 PSS5〜30 ms30〜200 ops/s最も一般的
    RSA-4096 PSS50〜300 ms3〜20 ops/sRSAは鍵長で激重
    ECDSA P-2562〜15 ms70〜500 ops/sRSAより軽い
    HMAC-SHA2560.5〜3 ms300〜2000 ops/s共通鍵で超軽い

    ※ ネットワーク型HSMの目安。HSMの世代、ネットワーク状態、データサイズで上下する

    設計上の落とし穴
    • 初回測定：TLSセッション確立、鍵キャッシュ展開で異常値が出る → ウォームアップ必須
    • セッション枯渇：マルチスレッドでセッションを使い回さないとHSM側上限に達する
    • 同一データ署名：同じデータで連続署名するとPSSの場合は毎回違う署名になる（saltがランダム）→ ベンチには影響しないが検証時に注意
    出典：[1] PKCS#11 Spec v3.2 §5.13[7] AWS CloudHSM Mechanisms[15] NIST FIPS 186-5

    CHAPTER 3 · 鍵生成 · 概要

    14鍵生成 — 属性テンプレートの世界

    PKCS#11でもっとも重い単発操作。Init/Finalなしの1関数で完結するが、属性テンプレートという概念が登場する。HSM内部で乱数生成→素数探索→検証→オブジェクト登録という重い処理が走る。

    ▸2つの関数

    C · 対称鍵 (AES等)
    CK_RV C_GenerateKey(
  CK_SESSION_HANDLE hSession,
  CK_MECHANISM_PTR  pMechanism,
  CK_ATTRIBUTE_PTR  pTemplate,
  CK_ULONG           ulCount,
  CK_OBJECT_HANDLE_PTR phKey
);
    C · 公開/秘密鍵ペア (RSA, EC)
    CK_RV C_GenerateKeyPair(

    対称鍵は本質的に「乱数取得+属性付け」なので軽い。非対称鍵は数学的に有効な鍵を探す必要があるため重い。

    なぜ「属性テンプレート」が必要か

    鍵を作るには、その鍵がどう使えるか（署名用？暗号化用？）、いつまで存在するか（永続？揮発？）、外に出せるか（持ち出し禁止？）等を生成時に決めてHSMに伝える必要がある。これらの設定が「属性テンプレート」という名前の配列として渡される。

    CHAPTER 3 · 鍵生成 · 準備

    15準備フェーズ — テンプレート構築

    鍵生成の準備の本体は属性テンプレートの構築。CK_ATTRIBUTE構造体の配列を作る作業で、ここでセキュリティ設計が決まる。

    ▸ PREPARATION

    属性テンプレートの構造

    テンプレートはCK_ATTRIBUTE構造体の配列。各要素は (属性タイプ, ポインタ, サイズ) の3つ組。

    RSA鍵ペア生成のテンプレート例。★が付いた属性がセキュリティの要
    セキュリティの要となる3属性
    • CKA_SENSITIVE = true：秘密値を C_GetAttributeValue で取り出せなくする
    • CKA_EXTRACTABLE = false：ラップしても外に出せない
    • CKA_PRIVATE = true：ログイン状態のユーザーにしか見えない

    この3つの組み合わせで「秘密鍵が絶対にHSMから出ない」状態を作る

    ベンダーごとの差
    • 属性のデフォルト値はベンダーによって違う
    • 必須属性も微妙に違う（CKA_IDなど）
    • 一部の属性は組み合わせ禁止（例：SENSITIVE=trueとEXTRACTABLE=trueを同時指定すると拒否されるベンダーあり）
    • ベンダーマニュアル必読

    CHAPTER 3 · 鍵生成 · アルゴリズム別特性

    17RSA / EC / AES — 全然違う3兄弟

    同じ「鍵生成」でも、アルゴリズムによって処理内容と所要時間が桁違いに変わる。性能評価ではこれらを別カテゴリで測るのが鉄則。

    ▸鍵長別の所要時間（典型値・ネットワーク型HSM）

    アルゴリズム / 鍵長meanp99max備考
    AES-256~1 ms~3 ms~10 msRTT支配
    EC P-2565〜10 ms~20 ms~40 ms安定
    EC P-38410〜30 ms~50 ms~100 ms安定
    RSA-2048200〜500 ms~2 sec数秒ばらつき大
    RSA-3072800ms〜2 sec~10 sec数十秒確率次第
    RSA-40962〜5 sec~30 sec分単位運要素強い
    なぜRSA鍵生成は遅いか

    RSA鍵生成 = 素数探索。たとえば2048ビットRSAなら1024ビットの素数を2つ探す。1024ビットの乱数1個が素数である確率は約 1/700。平均700回の素数判定試行が必要。Miller-Rabin判定自体も乱数を使う確率的処理で時間がかかる。

    出典：[16] NIST FIPS 186-5 §A.1.5 (RSA key gen)[1] PKCS#11 Spec v3.2 §6 (Mechanisms)

    C
    CK_MECHANISM mech = {
    CKM_RSA_PKCS_KEY_PAIR_GEN,
    NULL, 0
};

CK_ULONG  modBits = 2048;
CK_BYTE   pubExp[] = {0x01,0x00,0x01};
CK_BBOOL  yes = CK_TRUE;
CK_BBOOL  no  = CK_FALSE;

CK_ATTRIBUTE pubT[] = {
  {CKA_TOKEN,        &yes, sizeof(yes)},
  {CKA_MODULUS_BITS, &modBits, sizeof(modBits)},
  {CKA_PUBLIC_EXPONENT, pubExp, sizeof(pubExp)},
  {CKA_VERIFY,       &yes, sizeof(yes)},
};

CK_ATTRIBUTE privT[] = {
  {CKA_TOKEN,       &yes, sizeof(yes)},
  {CKA_PRIVATE,     &yes, sizeof(yes)},
  {CKA_SIGN,        &yes, sizeof(yes)},
  {CKA_SENSITIVE,   &yes, sizeof(yes)},
  {CKA_EXTRACTABLE, &no,  sizeof(no)},
};

CK_OBJECT_HANDLE hPub, hPriv;
CK_RV rv = pFuncs->C_GenerateKeyPair(
    hSession, &mech,
    pubT,  4,
    privT, 5,
    &hPub, &hPriv);

    CHAPTER 3 · 鍵生成 · ★ 性能評価

    19性能評価ポイント

    鍵生成の性能評価は署名とは完全に別ベンチで実施するのが鉄則。レイテンシのばらつきが激しく、署名と同じスケールで測ると統計が壊れる。

    ★ どこを測るか

    計測対象は C_GenerateKeyPair (or C_GenerateKey) の全体。Init/Finalなしの単発関数なので、計測点はシンプル。

    疑似コード
    // ★ 計測区間（テンプレート構築は事前に済ませる）
t0 = get_time_ns();
rv = pFuncs->C_GenerateKeyPair(
    hSession, &mech,
    pubT,  pubCount,
    privT, privCount,
    &hPub, &hPriv);
t1 = get_time_ns();

latency_ns = t1 - t0;
// 生成した鍵は使い回さない場合 C_DestroyObject で削除

    ▸測定すべき変数

    • アルゴリズム × 鍵長
      RSA: 2048/3072/4096
      EC: P-256/P-384/P-521
      AES: 128/192/256
    • 属性パターン：Token(永続) vs Session(揮発)
    • 並列度：注意 — HSM側の鍵生成同時実行数に上限あり

    ▸取るべきメトリクス（署名と違う！）

    • レイテンシ分布：max を必ず記録
    • p99 / p99.9：テールが長い
    • 標準偏差：ばらつきの大きさ
    • スループットより1リクエストの最悪値を重視
    設計上の重要な注意
    • 署名と同じ枠で測るな：1回の鍵生成 = 数百〜数千回の署名相当。平均が壊れる
    • HSM側の鍵格納上限：ベンチで作りすぎると枠を食う → 使い終わったら C_DestroyObject で削除
    • 並列度の罠：鍵生成はHSM側で並列実行されないことが多い（リソース集中型処理）。スレッドを増やしてもスループットが上がらない
    • サンプル数を稼ぐ：RSAは20-30回でmean計算するとmaxを取り逃がす。1000回以上のサンプリングが必要
    推奨ベンチ設計（RSA鍵生成）
    疑似コード
    // 1. 準備（測定外）
setup_session_and_login();

// 2. ウォームアップ（5〜10回）
for i in 0..10 { gen_and_destroy(); }

// 3. 計測 (1000サンプル / 数時間かかる)
for i in 0..1000 {
    t0 = now();
    C_GenerateKeyPair(...);
    t1 = now();
    latencies[i] = t1 - t0;
    C_DestroyObject(hPub);
    C_DestroyObject(hPriv);
}

// 4. 集計：max, p99.9 を必ず確認

    203操作の性能特性 — 横断比較

    乱数・署名・鍵生成は「単発の重さ」と「呼び出し頻度」が桁違い。ベンチ設計を間違えると、片方の特性しか見えなくなる。

    ▸性能特性の対比表

    観点 乱数生成 署名 鍵生成
    1回の所要時間 0.5〜数 ms 2〜30 ms (RSA-2048) 100ms〜数秒 (RSA-2048)
    ばらつき 小 中 超大（max≫mean）
    呼び出し頻度 中〜高 非常に高い 非常に低い
    主ボトルネック ネットワークRTT HSM内秘密鍵演算 HSM内素数探索
    並列スケール セッション数倍までOK HSM上限まで線形 ほぼスケールしない
    Init/Final なし（単発） あり（3段） なし（単発）
    計測単位 ops/sec ops/sec + p99 1回あたりmax
    サンプル数目安 10,000+ 1,000+ 100〜1,000（時間かかる）
    運用設計への含意
    • 鍵生成は事前に済ませる：本番リクエストの中で鍵生成しない（ユーザーを待たせる）
    • 署名のキャパが本番の上限：HSMの選定はTPS-署名を見る
    • 乱数は意外と詰まる：高負荷時はライブラリ側でPRNGを使うことも検討

    CLOSING · 総合まとめ

    21ベンチマーク設計の総合指針

    3操作を統一的に評価するための設計テンプレ。これに沿って組めば、結果の解釈が後で破綻しない。

    標準的なベンチマーク4フェーズ。3操作ともこの型で統一する
    必ず記録すべき情報
    • HSMのバージョン・FW・型番
    • PKCS#11ライブラリのバージョン
    • ネットワーク経路（同一LAN/別VPC等）
    • ベンチ実行ホストのスペック
    • 並列度・セッション数
    • 使用メカニズム・鍵長
    • テスト時刻（HSM側の他負荷影響）
    典型的な失敗パターン
    • ウォームアップなし → 初回がp99に化けて全体歪む
    • サンプル少なすぎ → maxを取り逃がし、テールリスク見落とし
    • 3操作を同じ枠で実行 → 鍵生成1回が他をブロック
    • セッション枯渇放置 → 中盤からエラー多発で結果汚染
    • HSM側ログを取り忘れ → ボトルネックの場所が特定できない
    ★ 並列度設計のチェックリスト
    • スレッドごとに別セッションを持つ（セッションを共有しない）
    • 並列度を1, 2, 4, 8, 16, 32, 64...と倍々で増やしてスケール曲線を描く
    • 飽和点を特定する：これ以上スレッドを増やしてもTPSが上がらない点
    • 飽和点でのレイテンシ・エラー率を記録（運用上限の参考）
    • 鍵生成は並列度1〜数で十分（スケールしないので）
