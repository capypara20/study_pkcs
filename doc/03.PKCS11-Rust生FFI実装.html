    03アーキテクチャ全体像 — 核心は C_GetFunctionList

    生FFIの肝は1点だけ。PKCS#11ライブラリがエクスポートする関数は実質 C_GetFunctionList ただ1つ。これを呼ぶと、全関数ポインタが詰まった構造体が返ってくる。

    シンボルを1個ずつ dlsym する必要はない。C_GetFunctionList が全部まとめて渡してくれる
    なぜこの設計なのか

    PKCS#11仕様が「C_GetFunctionList だけは必ずエクスポートせよ。他はオプション」と定めているため。バージョン差や関数の有無をこの構造体1つで吸収できる。

    cryptoki はこれを隠している

    高レベルの cryptoki クレートは、この C_GetFunctionList 呼び出しと構造体管理を内部でやってくれる。今回はその内部を自分で書く。

    出典：[1] PKCS#11 Spec v3.2 §5.4[3] libloading docs

    04ファイル構成案

    関心ごとに3層に分ける。FFI境界（unsafe）・ライフサイクル管理・操作実装。第2部の3操作は ops/ に1ファイルずつ対応させる。

    pkcs11-raw/ ├─ Cargo.toml # libloading 依存 └─ src/ ├─ main.rs # 全体配線・デモ実行 (slide 18) ├─ error.rs # CK_RV → Result<T, Error> (slide 12) ├─ loader.rs # ★ .so動的ロードの核心 (slide 13) ├─ session.rs # Init/Login/Session ライフサイクル (slide 14) ├─ ffi/ # ── unsafe な C 境界 ── │ ├─ mod.rs │ ├─ types.rs # CK_RV, CK_MECHANISM 等の型 (slide 9) │ ├─ function_list.rs # ★ CK_FUNCTION_LIST 構造体 (slide 10) │ └─ constants.rs # CKM_*, CKA_*, CKR_* 定数 (slide 11) └─ ops/ # ── 第2部の3操作に対応 ── ├─ mod.rs ├─ random.rs # Ch1 乱数生成 (slide 15) ├─ sign.rs # Ch2 署名 (slide 16) └─ keygen.rs # Ch3 鍵生成 (slide 17)
    第1層：FFI境界

    ffi/ 配下。C構造体・型・定数を#[repr(C)]で正確に写す。unsafeはここに閉じ込める。

    第2層：管理

    loader.rs / session.rs。.soのロード・解放、セッションの開閉をRAIIで安全に包む。

    第3層：操作

    ops/配下。乱数・署名・鍵生成を安全な公開APIとして提供。第2部のフローと1対1。

    05モジュール依存図

    依存は一方向。上位（ops）が下位（ffi）に依存し、逆流しない。unsafeは下に行くほど濃く、上に行くほど安全になる。

    上位ほど安全なAPI、下位ほど生のC境界。依存は上から下への一方向
    設計の狙い
    • unsafeの局所化：ffi/ と loader.rs 以外に unsafe を書かせない
    • テスト容易性：ops/ は Session さえあれば単体で扱える
    • 第2部との対応：ops/{random,sign,keygen}.rs がフロー図と1対1

    06型設計案 — C型を Rust にどう写すか

    PKCS#11のC型を#[repr(C)]でバイト配置を一致させる。整数は c_ulong 系、構造体はフィールド順を厳守。ここがズレると即UB。

    ▸スカラ型のマッピング

    PKCS#11 (C)Rust
    CK_BYTEu8
    CK_BBOOLu8 (0/1)
    CK_ULONGc_ulong
    CK_RVc_ulong
    CK_SESSION_HANDLEc_ulong
    CK_OBJECT_HANDLEc_ulong
    CK_xxx_PTR*mut T

    ※ CK_ULONG はプラットフォームのlong幅依存。c_ulongを使えば自動で合う

    ▸構造体（フィールド順厳守）

    Rust
    #[repr(C)]
pub struct CK_MECHANISM {
    pub mechanism:     CK_MECHANISM_TYPE,
    pub pParameter:    *mut c_void,
    pub ulParameterLen:CK_ULONG,
}

#[repr(C)]
pub struct CK_ATTRIBUTE {
    pub type_:      CK_ATTRIBUTE_TYPE,
    pub pValue:     *mut c_void,
    pub ulValueLen: CK_ULONG,
}
    設計上の決定事項
    • 命名はC流を維持：CK_MECHANISM 等を #[allow(non_camel_case_types)] でそのまま使う。仕様書と突き合わせやすくするため（第2部の図とも一致）
    • 関数ポインタは Option<unsafe extern "C" fn(...)>：CではNULLになり得るため、OptionでNULL安全に表現
    • ハンドルは透過的：CK_SESSION_HANDLE 等は単なる整数だが、newtypeで包むかは好み。今回は素直に整数のまま
    出典：[1] PKCS#11 Spec v3.2 §3-4 (型定義)[7] Rustonomicon (repr(C))

    07関数シグネチャ案 — 2つのレイヤー

    シグネチャは2層で考える。下層は生のFFI関数ポインタ型（C仕様そのまま）。上層はそれを包んだ安全な公開API（Rust流）。

    ▸下層：FFI関数ポインタ型

    Rust · ffi
    // C_GetFunctionList のシグネチャ
type CGetFunctionList = unsafe extern "C" fn(
    *mut *mut CK_FUNCTION_LIST,
) -> CK_RV;

// 構造体内の関数ポインタ（例）
pub C_GenerateRandom: Option<unsafe extern "C" fn(
    CK_SESSION_HANDLE,
    *mut CK_BYTE,   // 出力バッファ
    CK_ULONG,        // 長さ
) -> CK_RV>,

    C仕様の関数プロトタイプをそのまま型にする。引数の順序・型を厳守

    ▸上層：安全な公開API

    Rust · 公開API
    impl Module {
    pub fn load(path: &str)
        -> Result<Module, Error>;
    pub fn open_session(&self, slot: CK_SLOT_ID)
        -> Result<Session, Error>;
}

impl Session<'_> {
    // Ch1
    pub fn generate_random(&self, n: usize)
        -> Result<Vec<u8>, Error>;
    // Ch2
    pub fn sign(&self, m: CK_MECHANISM_TYPE,
        key: CK_OBJECT_HANDLE, data: &[u8])
        -> Result<Vec<u8>, Error>;
    // Ch3
    pub fn generate_rsa_keypair(&self,
        bits: CK_ULONG)
        -> Result<(CK_OBJECT_HANDLE,
                  CK_OBJECT_HANDLE), Error>;
}
    設計方針：生のCを上層に漏らさない

    下層は *mut ポインタと CK_RV 整数まみれ。上層ではこれらを Vec<u8> や Result に変換して返す。呼び出し側（main.rs）は unsafe を一切書かずに済む。これがcryptokiがやっていることの再現。

    09ffi/types.rs — 基本型

    C型を#[repr(C)]でRustへマッピングしたFFI境界の土台。設計案（slide 06）をそのままコードに落とす。

    src/ffi/types.rs
    Rust
    #![allow(non_camel_case_types)]
use std::os::raw::{c_uchar, c_ulong, c_void};

// ── スカラ型（C仕様のtypedefを忠実に）──
pub type CK_BYTE            = c_uchar;
pub type CK_BBOOL           = c_uchar;   // 0 = false, それ以外 = true
pub type CK_ULONG           = c_ulong;
pub type CK_RV              = c_ulong;   // 戻り値コード
pub type CK_FLAGS           = c_ulong;
pub type CK_SLOT_ID         = c_ulong;
pub type CK_SESSION_HANDLE  = c_ulong;   // 第2部の「セッション」
pub type CK_OBJECT_HANDLE   = c_ulong;   // 第2部の「預かり証」=鍵ハンドル
pub type CK_USER_TYPE       = c_ulong;
pub type CK_MECHANISM_TYPE  = c_ulong;   // CKM_*
pub type CK_ATTRIBUTE_TYPE  = c_ulong;   // CKA_*

// ── バージョン構造体 ──
#[repr(C)]
#[derive(Debug, Clone, Copy)]
pub struct CK_VERSION {
    pub major: CK_BYTE,
    pub minor: CK_BYTE,
}

// ── メカニズム（署名・鍵生成で「方式」を指定）──
#[repr(C)]
pub struct CK_MECHANISM {
    pub mechanism:      CK_MECHANISM_TYPE,
    pub pParameter:     *mut c_void,   // PSS等の追加パラメータ。無ければ null
    pub ulParameterLen: CK_ULONG,
}

// ── 属性（鍵生成のテンプレート要素）──
#[repr(C)]
pub struct CK_ATTRIBUTE {
    pub type_:      CK_ATTRIBUTE_TYPE,  // CKA_TOKEN 等
    pub pValue:     *mut c_void,        // 値へのポインタ
    pub ulValueLen: CK_ULONG,           // 値のバイト長
}

// セッションコールバック（今回は使わないので常に None を渡す）
pub type CK_NOTIFY = Option<
    unsafe extern "C" fn(CK_SESSION_HANDLE, CK_ULONG, *mut c_void) -> CK_RV
>;

    10ffi/function_list.rs — ★ 順序の罠

    最大の落とし穴がここ。CK_FUNCTION_LISTの各フィールドは仕様で定められた順序で全関数を並べる必要がある。順番を間違えると別の関数を呼んでしまう。

    ⚠ なぜ順序が致命的か

    この構造体はメモリ上で関数ポインタが連続して並んだ配列のようなもの。C_Signを呼んだつもりが、フィールドを1つ飛ばしていたら隣のC_SignUpdateを呼ぶことになる。コンパイルは通り、実行時に謎の挙動になる。必要な関数だけ抜き出して書いてはいけない。

    src/ffi/function_list.rs（抜粋：先頭から仕様順に）
    Rust
    #![allow(non_snake_case)]
use std::os::raw::c_void;
use super::types::*;

// 関数ポインタ型のエイリアス（よく使う形）
type Fn1<A>        = Option<unsafe extern "C" fn(A) -> CK_RV>;

#[repr(C)]
pub struct CK_FUNCTION_LIST {
    pub version: CK_VERSION,        // ← 必ず先頭

    // ↓↓↓ ここからは PKCS#11 仕様の C_xxx 出現順を厳守 ↓↓↓
    pub C_Initialize:   Fn1<*mut c_void>,                 // 1
    pub C_Finalize:     Fn1<*mut c_void>,                 // 2
    pub C_GetInfo:      Fn1<*mut c_void>,                 // 3
    pub C_GetFunctionList: Fn1<*mut c_void>,              // 4
    pub C_GetSlotList:  Option<unsafe extern "C" fn(        // 5
        CK_BBOOL, *mut CK_SLOT_ID, *mut CK_ULONG) -> CK_RV>,
    // ... 6〜12 は GetSlotInfo/GetTokenInfo/GetMechanism.../InitToken/InitPIN/SetPIN ...

    pub C_OpenSession:  Option<unsafe extern "C" fn(        // 13
        CK_SLOT_ID, CK_FLAGS, *mut c_void,
        CK_NOTIFY, *mut CK_SESSION_HANDLE) -> CK_RV>,
    pub C_CloseSession: Fn1<CK_SESSION_HANDLE>,            // 14
    // ... 15〜18 ...
    pub C_Login:        Option<unsafe extern "C" fn(        // 19
        CK_SESSION_HANDLE, CK_USER_TYPE,
        *mut CK_BYTE, CK_ULONG) -> CK_RV>,
    pub C_Logout:       Fn1<CK_SESSION_HANDLE>,            // 20
    // ... 21〜42 (オブジェクト操作・暗号・ダイジェスト) ...

    pub C_SignInit:     Option<unsafe extern "C" fn(        // 43
        CK_SESSION_HANDLE, *mut CK_MECHANISM, CK_OBJECT_HANDLE) -> CK_RV>,
    pub C_Sign:         Option<unsafe extern "C" fn(        // 44
        CK_SESSION_HANDLE, *mut CK_BYTE, CK_ULONG,
        *mut CK_BYTE, *mut CK_ULONG) -> CK_RV>,
    // ... 45〜58 (SignUpdate/Final, Verify系, 複合系) ...

    pub C_GenerateKey:  Option<unsafe extern "C" fn(        // 59
        CK_SESSION_HANDLE, *mut CK_MECHANISM,
        *mut CK_ATTRIBUTE, CK_ULONG, *mut CK_OBJECT_HANDLE) -> CK_RV>,
    pub C_GenerateKeyPair: Option<unsafe extern "C" fn(     // 60
        CK_SESSION_HANDLE, *mut CK_MECHANISM,
        *mut CK_ATTRIBUTE, CK_ULONG,    // 公開鍵テンプレート
        *mut CK_ATTRIBUTE, CK_ULONG,    // 秘密鍵テンプレート
        *mut CK_OBJECT_HANDLE, *mut CK_OBJECT_HANDLE) -> CK_RV>,
    // ... 61〜64 (WrapKey/UnwrapKey/DeriveKey/SeedRandom) ...

    pub C_GenerateRandom: Option<unsafe extern "C" fn(      // 65
        CK_SESSION_HANDLE, *mut CK_BYTE, CK_ULONG) -> CK_RV>,
    // ... 66〜68 (GetFunctionStatus/CancelFunction/WaitForSlotEvent) で終端 ...
}

    11ffi/constants.rs — 定数

    第2部で登場した CKR_*（戻り値）・CKM_*（メカニズム）・CKA_*（属性）・CKF_*（フラグ）を数値で定義。値はPKCS#11仕様で固定されている。

    src/ffi/constants.rs
    Rust

use super::types::{CK_ULONG, CK_FLAGS, CK_USER_TYPE};

// ── 戻り値 CK_RV（第2部「結果フェーズ」で頻出）──
pub const CKR_OK:                  CK_ULONG = 0x0000_0000;
pub const CKR_ARGUMENTS_BAD:       CK_ULONG = 0x0000_0007;
pub const CKR_SESSION_HANDLE_INVALID: CK_ULONG = 0x0000_00B3;

// ── セッションフラグ（OpenSession で使用）──
pub const CKF_RW_SESSION:     CK_FLAGS = 0x0000_0002;
pub const CKF_SERIAL_SESSION: CK_FLAGS = 0x0000_0004;  // 必須

// ── ユーザー種別（Login で使用）──
pub const CKU_USER: CK_USER_TYPE = 1;

// ── メカニズム CKM_*（署名・鍵生成の「方式」）──
pub const CKM_RSA_PKCS_KEY_PAIR_GEN: CK_ULONG = 0x0000_0000;  // Ch3 RSA鍵生成
pub const CKM_RSA_PKCS:              CK_ULONG = 0x0000_0001;
pub const CKM_SHA256_RSA_PKCS:       CK_ULONG = 0x0000_0040;  // Ch2 署名
pub const CKM_AES_KEY_GEN:           CK_ULONG = 0x0000_1080;

// ── 属性 CKA_*（鍵生成テンプレートのキー。第2部 Ch3 の図と一致）──
pub const CKA_CLASS:           CK_ULONG = 0x0000_0000;
pub const CKA_TOKEN:           CK_ULONG = 0x0000_0001;  // 永続化
pub const CKA_PRIVATE:         CK_ULONG = 0x0000_0002;
pub const CKA_KEY_TYPE:        CK_ULONG = 0x0000_0100;
pub const CKA_SENSITIVE:       CK_ULONG = 0x0000_0103;  // ★秘密値保護
pub const CKA_SIGN:            CK_ULONG = 0x0000_0108;  // 署名可
pub const CKA_VERIFY:          CK_ULONG = 0x0000_010A;  // 検証可
pub const CKA_MODULUS_BITS:    CK_ULONG = 0x0000_0121;  // RSA鍵長
pub const CKA_PUBLIC_EXPONENT: CK_ULONG = 0x0000_0122;
pub const CKA_EXTRACTABLE:     CK_ULONG = 0x0000_0162;  // ★持ち出し可否

pub const CK_TRUE:  u8 = 1;
pub const CK_FALSE: u8 = 0;

    これらの16進値はPKCS#11ヘッダ（pkcs11t.h）で定義された固定値。勝手に変えてはいけない。ベンダー拡張はCKM_VENDOR_DEFINED(0x8000_0000)以降の領域を使う。

    12error.rs — CK_RV を Result に変える

    C流の「戻り値で成否を返す」を、Rust流の Result に変換する境界。これで上層は ? 演算子で素直にエラー伝播できる。

    src/error.rs
    Rust
    use crate::ffi::types::CK_RV;
use crate::ffi::constants::CKR_OK;

#[derive(Debug)]
pub enum Error {
    // .so のロード失敗（libloading由来）
    Load(libloading::Error),
    // C_GetFunctionList が NULL を返した
    NullFunctionList,
    // その関数ポインタが NULL（= 未実装）だった
    FunctionNotSupported(&'static str),
    // PKCS#11 がエラーコードを返した（CK_RV != CKR_OK）
    Pkcs11(CK_RV),
}

// libloading::Error を自動変換（? で使えるように）
impl From<libloading::Error> for Error {
    fn from(e: libloading::Error) -> Self { Error::Load(e) }
}

// CK_RV をチェックして Result に変換する中核ヘルパ
#[inline]
pub fn check(rv: CK_RV) -> Result<(), Error> {
    if rv == CKR_OK {
        Ok(())
    } else {
        Err(Error::Pkcs11(rv))
    }
}

// Option<fn> を取り出すヘルパ（NULL関数ポインタ対策）
pub fn require<F>(f: Option<F>, name: &'static str) -> Result<F, Error> {
    f.ok_or(Error::FunctionNotSupported(name))
}

impl std::fmt::Display for Error {
    fn fmt(&self, f: &mut std::fmt::Formatter) -> std::fmt::Result {
        match self {
            Error::Load(e)            => write!(f, "load error: {e}"),
            Error::NullFunctionList    => write!(f, "C_GetFunctionList returned null"),
            Error::FunctionNotSupported(n) => write!(f, "function not supported: {n}"),
            Error::Pkcs11(rv)         => write!(f, "PKCS#11 error: 0x{rv:08x}"),
        }
    }
}
impl std::error::Error for Error {}
    この変換が「安全な層」の入口

    check(rv)? と書けば、エラーコードが自動で Error::Pkcs11 に化けて伝播する。第2部で挙げた CKR_SESSION_HANDLE_INVALID 等は、すべてこの Error::Pkcs11(0x...) に集約される。

    出典：[1] PKCS#11 Spec v3.2 §5.1 (戻り値)[6] Rust API Guidelines (C-GOOD-ERR)

    13loader.rs — ★ .so動的ロードの核心

    このドキュメントの主役。libloadingで.soを開き、C_GetFunctionListを1回だけ呼んで、関数ポインタ束を手に入れる。slide 03の図がそのままコードになる。

    src/loader.rs
    Rust
    use libloading::{Library, Symbol};
use std::ptr;
use crate::ffi::function_list::CK_FUNCTION_LIST;
use crate::ffi::types::CK_RV;
use crate::error::{check, Error};

// C_GetFunctionList のシグネチャ（唯一エクスポートされるシンボル）
type CGetFunctionList =
    unsafe extern "C" fn(*mut *mut CK_FUNCTION_LIST) -> CK_RV;

pub struct Module {
    // ★最重要：Library を保持し続ける。drop されると .so がアンロード(dlclose)され、
    //          関数ポインタが全てダングリングになる。順序も drop 順に効く。
    _lib: Library,
    // .so 内部の静的構造体を指すポインタ（所有はしない）
    funcs: *const CK_FUNCTION_LIST,
}

impl Module {
    // 指定パスの .so をロードし、関数リストを取得する
    pub fn load(path: &str) -> Result<Module, Error> {
        unsafe {
            // (1) .so を開く（dlopen 相当）
            let lib = Library::new(path)?;

            // (2) C_GetFunctionList シンボルを取得（唯一必要な dlsym）
            let entry: Symbol<CGetFunctionList> =
                lib.get(b"C_GetFunctionList")?;

            // (3) 呼び出して関数リストへのポインタを受け取る
            let mut list: *mut CK_FUNCTION_LIST = ptr::null_mut();
            let rv = entry(&mut list);
            check(rv)?;
            if list.is_null() {
                return Err(Error::NullFunctionList);
            }

            // (4) Library を一緒に保持して返す（ここが寿命管理の肝）
            Ok(Module { _lib: lib, funcs: list })
        }
    }

    // 関数リストへの安全な参照を返す内部ヘルパ
    #[inline]
    pub(crate) fn funcs(&self) -> &CK_FUNCTION_LIST {
        // load() で null チェック済みなので参照化は安全
        unsafe { &*self.funcs }
    }
}

// .so ロード時点では C_Initialize はまだ呼んでいない。
// → Initialize / Finalize は session.rs 側で扱う（次スライド）
    ⚠ Library を捨ててはいけない

    よくあるバグ：C_GetFunctionListでポインタだけ取ってLibraryをスコープ外に落とす。すると.soがアンロードされ、後で関数を呼んだ瞬間にSEGV。_libフィールドで構造体の寿命に縛り付けるのが正解。

    14session.rs — ライフサイクル管理

    第2部の14ステップ・ライフサイクルのうち、Initialize → OpenSession → Login → (操作) → CloseSession → Finalize を RAII で安全に包む。Dropで後始末を自動化。

    src/session.rs
    Rust
    use std::ptr;
use crate::loader::Module;
use crate::ffi::types::*;
use crate::ffi::constants::*;
use crate::error::{check, require, Error};

impl Module {
    // ライブラリ全体の初期化（第2部 step: C_Initialize）
    pub fn initialize(&self) -> Result<(), Error> {
        let f = require(self.funcs().C_Initialize, "C_Initialize")?;
        // pInitArgs は NULL でOK（シングルスレッド既定）
        check(unsafe { f(ptr::null_mut()) })
    }

    // 最初の「トークン入りスロット」を取得
    pub fn first_slot(&self) -> Result<CK_SLOT_ID, Error> {
        let f = require(self.funcs().C_GetSlotList, "C_GetSlotList")?;
        let mut count: CK_ULONG = 0;
        // 1回目：個数だけ取得（2-call convention）
        check(unsafe { f(CK_TRUE, ptr::null_mut(), &mut count) })?;
        let mut slots = vec![0 as CK_SLOT_ID; count as usize];
        // 2回目：実データ取得
        check(unsafe { f(CK_TRUE, slots.as_mut_ptr(), &mut count) })?;
        slots.first().copied().ok_or(Error::Pkcs11(0))
    }

    // セッションを開く（第2部 step: C_OpenSession）
    pub fn open_session(&self, slot: CK_SLOT_ID) -> Result<Session, Error> {
        let f = require(self.funcs().C_OpenSession, "C_OpenSession")?;
        let mut h: CK_SESSION_HANDLE = 0;
        let flags = CKF_SERIAL_SESSION | CKF_RW_SESSION;
        check(unsafe {
            f(slot, flags, ptr::null_mut(), None, &mut h)
        })?;
        Ok(Session { module: self, handle: h })
    }
}

// セッションは Module を借用する（'a で寿命を縛る）
pub struct Session<'a> {
    pub(crate) module: &'a Module,
    pub(crate) handle: CK_SESSION_HANDLE,
}

impl Session<'_> {
    // ログイン（第2部 step: C_Login）
    pub fn login(&self, pin: &str) -> Result<(), Error> {
        let f = require(self.module.funcs().C_Login, "C_Login")?;
        let pin_bytes = pin.as_bytes();
        check(unsafe {
            f(self.handle, CKU_USER,
              pin_bytes.as_ptr() as *mut CK_BYTE,
              pin_bytes.len() as CK_ULONG)
        })
    }
}

// ★ RAII：Session が drop されると自動で CloseSession
impl Drop for Session<'_> {
    fn drop(&mut self) {
        if let Some(f) = self.module.funcs().C_CloseSession {
            unsafe { f(self.handle); }  // エラーは握り潰す（drop中なので）
        }
    }
}

    15ops/random.rs 第2部 Ch1 対応

    第2部Chapter 1の乱数生成フローを生FFIで実装。Init/Finalなしの単発呼び出し。バッファをRust側で確保し、ポインタを渡すだけ。

    src/ops/random.rs
    Rust
    use crate::session::Session;
use crate::ffi::types::*;
use crate::error::{check, require, Error};

impl Session<'_> {
    /// 乱数生成（第2部 Ch1）。C_GenerateRandom を1回呼ぶだけ。
    pub fn generate_random(&self, len: usize) -> Result<Vec<u8>, Error> {
        // (準備) 出力バッファを Rust 側で確保
        let mut buf = vec![0u8; len];

        // 関数ポインタを取得（NULL なら未サポート）
        let f = require(self.module.funcs().C_GenerateRandom, "C_GenerateRandom")?;

        // (実行) HSM内のTRNGからバイト列を取得
        let rv = unsafe {
            f(
                self.handle,
                buf.as_mut_ptr(),          // *mut CK_BYTE
                buf.len() as CK_ULONG,     // 長さ
            )
        };

        // (結果) CK_RV を Result に変換して返す
        check(rv)?;
        Ok(buf)
    }
}
    第2部フローとの対応
    • 準備vec![0u8; len] でバッファ確保
    • 実行f(handle, ptr, len) でHSMのTRNG呼び出し
    • 結果check(rv)? で成否判定、Vec<u8> を返す
    生FFIの安全化ポイント

    unsafeブロックは関数呼び出しの1行だけ。バッファ確保も結果変換も安全なRustコード。呼び出し側はsession.generate_random(32)?と書くだけで、ポインタもunsafeも見えない。

    16ops/sign.rs 第2部 Ch2 対応

    第2部Chapter 2の署名フローを実装。C_SignInit → C_Sign の2段。さらに2-call convention（署名長を先に問い合わせ）も再現する。

    src/ops/sign.rs
    Rust
    use std::ptr;
use crate::session::Session;
use crate::ffi::types::*;
use crate::error::{check, require, Error};

impl Session<'_> {
    /// 署名（第2部 Ch2）。SignInit → Sign(長さ取得) → Sign(本番) の流れ。
    pub fn sign(
        &self,
        mech: CK_MECHANISM_TYPE,
        key: CK_OBJECT_HANDLE,
        data: &[u8],
    ) -> Result<Vec<u8>, Error> {
        // (準備) メカニズム構造体を組む。パラメータ無しの方式を想定
        let mut mechanism = CK_MECHANISM {
            mechanism: mech,
            pParameter: ptr::null_mut(),
            ulParameterLen: 0,
        };

        // ── C_SignInit：このセッションを「署名待ち」状態に ──
        let sign_init = require(self.module.funcs().C_SignInit, "C_SignInit")?;
        check(unsafe {
            sign_init(self.handle, &mut mechanism, key)
        })?;

        let sign = require(self.module.funcs().C_Sign, "C_Sign")?;
        let data_ptr = data.as_ptr() as *mut CK_BYTE;
        let data_len = data.len() as CK_ULONG;

        // ── 1回目：pSignature = NULL で署名長を問い合わせ ──
        let mut sig_len: CK_ULONG = 0;
        check(unsafe {
            sign(self.handle, data_ptr, data_len,
                 ptr::null_mut(), &mut sig_len)
        })?;

        // ── 2回目：確定した長さでバッファを確保して本番署名 ──
        let mut sig = vec![0u8; sig_len as usize];
        check(unsafe {
            sign(self.handle, data_ptr, data_len,
                 sig.as_mut_ptr(), &mut sig_len)
        })?;

        // HSMが返した実際の長さに切り詰める
        sig.truncate(sig_len as usize);
        Ok(sig)
        // ※ C_Sign 完了でセッション状態は自動リセット（第2部参照）
    }
}
    2-call convention を省略する場合

    RSA-2048なら署名長は常に256バイトと分かっているので、1回目を省いて固定サイズバッファで呼んでも動く。ただし鍵長やメカニズムが可変なら2回呼びが安全。第2部で触れた通り。

    17ops/keygen.rs 第2部 Ch3 対応

    第2部Chapter 3の鍵生成フローを実装。属性テンプレート配列を組んで C_GenerateKeyPair に渡す。slide 15（第2部）の公開鍵/秘密鍵テンプレート図がそのままコードになる。

    src/ops/keygen.rs
    Rust
    use std::os::raw::c_void;
use std::ptr;
use crate::session::Session;
use crate::ffi::types::*;
use crate::ffi::constants::*;
use crate::error::{check, require, Error};

// 属性を1つ作るヘルパ（CK_ATTRIBUTE の組み立てを短く）
fn attr(t: CK_ATTRIBUTE_TYPE, val: *mut c_void, len: usize) -> CK_ATTRIBUTE {
    CK_ATTRIBUTE { type_: t, pValue: val, ulValueLen: len as CK_ULONG }
}

impl Session<'_> {
    /// RSA鍵ペア生成（第2部 Ch3）。(公開鍵ハンドル, 秘密鍵ハンドル) を返す。
    pub fn generate_rsa_keypair(
        &self,
        modulus_bits: CK_ULONG,
    ) -> Result<(CK_OBJECT_HANDLE, CK_OBJECT_HANDLE), Error> {
        // (準備1) メカニズム = RSA鍵ペア生成
        let mut mech = CK_MECHANISM {
            mechanism: CKM_RSA_PKCS_KEY_PAIR_GEN,
            pParameter: ptr::null_mut(),
            ulParameterLen: 0,
        };

        // (準備2) 属性値の実体。HSMは読むだけなので true は共有してよい
        let mut bits = modulus_bits;
        let mut pub_exp: [u8; 3] = [0x01, 0x00, 0x01];  // 65537
        let mut yes = CK_TRUE;
        let mut no  = CK_FALSE;
        let b = |p: &mut u8| p as *mut _ as *mut c_void;

        // (準備3) 公開鍵テンプレート（第2部 slide15 左側）
        let mut pub_tpl = [
            attr(CKA_TOKEN,           b(&mut yes), 1),
            attr(CKA_MODULUS_BITS,    &mut bits as *mut _ as *mut c_void,
                                     std::mem::size_of::<CK_ULONG>()),
            attr(CKA_PUBLIC_EXPONENT, pub_exp.as_mut_ptr() as *mut c_void, 3),
            attr(CKA_VERIFY,          b(&mut yes), 1),
        ];

        // (準備4) 秘密鍵テンプレート（第2部 slide15 右側。★が要）
        let mut priv_tpl = [
            attr(CKA_TOKEN,       b(&mut yes), 1),
            attr(CKA_PRIVATE,     b(&mut yes), 1),
            attr(CKA_SIGN,        b(&mut yes), 1),
            attr(CKA_SENSITIVE,   b(&mut yes), 1),  // ★秘密値保護
            attr(CKA_EXTRACTABLE, b(&mut no),  1),  // ★持ち出し禁止
        ];

        // (実行) HSM内で素数探索 → 鍵ペア構築 → 2つのハンドルを受け取る
        let mut h_pub: CK_OBJECT_HANDLE = 0;
        let mut h_priv: CK_OBJECT_HANDLE = 0;
        let gen = require(self.module.funcs().C_GenerateKeyPair, "C_GenerateKeyPair")?;
        check(unsafe {
            gen(
                self.handle,
                &mut mech,
                pub_tpl.as_mut_ptr(),  pub_tpl.len() as CK_ULONG,
                priv_tpl.as_mut_ptr(), priv_tpl.len() as CK_ULONG,
                &mut h_pub, &mut h_priv,
            )
        })?;

        Ok((h_pub, h_priv))
    }
}

    18main.rs — 全体配線

    全部つなぐ。ロード → 初期化 → スロット選択 → セッション → ログイン → 3操作。第2部の14ステップ・ライフサイクルが、ここで1本の流れになる。

    src/main.rs
    Rust
    mod error;
mod loader;
mod session;
mod ffi;
mod ops;

use loader::Module;
use ffi::constants::CKM_SHA256_RSA_PKCS;

fn main() -> Result<(), Box<dyn std::error::Error>> {
    // .so のパスは環境変数で（既定は SoftHSM2）
    let path = std::env::var("PKCS11_MODULE")
        .unwrap_or_else(|_| "/usr/lib/softhsm/libsofthsm2.so".into());

    // (1) .so を動的ロードして関数リスト取得（slide 13）
    let module = Module::load(&path)?;
    println!("loaded: {path}");

    // (2) ライブラリ初期化（C_Initialize）
    module.initialize()?;

    // (3) トークン入りスロットを選ぶ
    let slot = module.first_slot()?;
    println!("slot: {slot}");

    // (4) セッション開設 + ログイン
    let session = module.open_session(slot)?;
    session.login("1234")?;

    // ───── 第2部の3操作を実行 ─────

    // Ch1: 乱数生成
    let random = session.generate_random(32)?;
    println!("random (32B): {:02x?}", &random[..8]);

    // Ch3: RSA-2048 鍵ペア生成（署名に使う鍵を先に作る）
    let (pub_key, priv_key) = session.generate_rsa_keypair(2048)?;
    println!("keypair: pub={pub_key}, priv={priv_key}");

    // Ch2: 署名
    let sig = session.sign(CKM_SHA256_RSA_PKCS, priv_key, b"hello pkcs11")?;
    println!("signature: {} bytes", sig.len());

    Ok(())
    // session が drop → C_CloseSession 自動実行（slide 14）
    // module が drop → .so アンロード
}

    19生FFIの落とし穴と対策

    cryptokiが隠してくれている危険を、自分で書くと全部踏みうる。コンパイルが通っても実行時に壊れる類いが多いのが厄介。要点を5つ。

    ① Library の寿命

    Libraryをdropすると.soがアンロードされ、関数ポインタが全部ダングリング。→ 必ず構造体に_libとして保持（slide 13）。

    ② CK_FUNCTION_LIST の順序

    フィールドを1つ飛ばすと隣の関数を呼ぶ。コンパイルは通る。→ 全関数を仕様順に書くかcryptoki-sys/bindgenで生成（slide 10）。

    ③ 関数ポインタのNULL

    未実装の関数はNULL。Optionのままunwrapせずrequire()で明示的にチェック（slide 12）。

    ④ repr(C) とフィールド順

    #[repr(C)]を付け忘れるとRustがフィールドを並べ替え、レイアウトがCと不一致に。構造体の定義順も型もC仕様と完全一致させる（slide 06/09）。

    ⑤ drop順序と二重解放

    Sessionより先にModuleがdropされると、C_CloseSessionが死んだ.soを叩く。→ 借用&'a Moduleで寿命を縛り、Sessionが先にdropされることを型で保証（slide 14）。

    実務の結論

    学習・検証には生FFIで内部理解。本番はcryptoki（高レベル）を使うのが安全。今回のコードは「cryptokiの中身」を自分で書いたもの。

    第2部（性能評価）との接続

    性能測定をやるなら、このops/の各関数をstd::time::Instantで挟むだけ。ただし第2部で述べた通り、鍵生成は署名と時間スケールが違うので別ベンチにする・ウォームアップを入れるのを忘れずに。生FFIなら測定オーバーヘッドが最小なのも利点。
