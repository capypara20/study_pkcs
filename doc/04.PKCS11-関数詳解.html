    CHAPTER 1 · 暗号化・復号 · 概要

    03暗号化・復号 — AES-GCM の構造を理解する

    署名が「秘密鍵操作」だったのに対し、暗号化は対称鍵（AES等）またはRSA公開鍵で行う。PKCS#11の暗号化APIも3段ロケット構造だが、IVや認証タグ（GCM）をどう扱うかがポイント。

    暗号化関数の3段ロケット
    • ①
      C_EncryptInit
      セッションに「どの鍵で・どのモードで」を宣言
    • ②
      C_Encrypt / C_EncryptUpdate
      データを渡して暗号文を取得
    • ③
      C_EncryptFinal (分割時のみ)
      最終ブロックと認証タグを受け取る
    GCM vs CBC の違い
    項目AES-GCMAES-CBC
    認証タグあり（改ざん検知）なし
    パディング不要PKCS#7必要
    並列化可能不可（前ブロック依存）
    MechanismCKM_AES_GCMCKM_AES_CBC_PAD
    出典：[1] PKCS#11 Spec v3.2 §5.14[2] PKCS#11 Spec v3.2 §2.11 AES-GCM[11] NIST SP 800-38D

    CHAPTER 1 · 暗号化 · シーケンス

    04C_EncryptInit → C_Encrypt — シーケンス図

    C_EncryptInit で「鍵＋メカニズム」を宣言し、C_Encrypt でデータを渡す。AES-GCMの場合、HSM内部でIVを生成し、暗号文と認証タグを合成して返す（またはアプリがIVを指定）。

    2-call convention（NULL渡し→サイズ取得→実暗号化）はSignと同じパターン
    GCMパラメータ構造体
    C / PKCS#11
    CK_GCM_PARAMS gcm = {
    .pIv       = iv_bytes,    /* C_GenerateRandom で取得した12B */
    .ulIvLen   = 12,
    .pAAD      = aad_bytes,   /* NULL可（任意の追加認証データ）*/
    .ulAADLen  = aad_len,
    .ulTagBits = 128,        /* 認証タグは16B(128bit)が推奨 */
};
CK_MECHANISM mech = { CKM_AES_GCM, &gcm, sizeof(gcm) };
    出典：[1] PKCS#11 Spec v3.2 §5.14.1, §2.11.1

    CHAPTER 1 · 復号 · コード例

    05C_Decrypt — コード例とモード比較

    復号は暗号化の逆順。GCMでは認証タグの検証に失敗すると CKR_ENCRYPTED_DATA_INVALID が返る。これが改ざん検知の要。

    ▸C 言語 — AES-GCM 復号

    C
    /* --- 復号 (AES-GCM) --- */
CK_GCM_PARAMS gcm = {
    .pIv=iv, .ulIvLen=12,
    .pAAD=aad, .ulAADLen=aad_len,
    .ulTagBits=128
};
CK_MECHANISM mech = { CKM_AES_GCM, &gcm, sizeof(gcm) };

C_DecryptInit(sess, &mech, hAesKey);

/* ciphertext = 暗号文 + タグ(16B) */
CK_ULONG plainLen = cipher_len - 16;
CK_RV rv = pF->C_Decrypt(
    sess, cipher, cipher_len,
    plain_buf, &plainLen);

if (rv == CKR_ENCRYPTED_DATA_INVALID) {
    puts("⚠ タグ不一致：改ざんを検知");
} else if (rv == CKR_OK) {
    /* plain_buf[0..plainLen] が復号結果 */
}

    ▸Rust (cryptoki)

    Rust
    use cryptoki::mechanism::{
    Mechanism, aes::GcmParams
};

let gcm = GcmParams::new(
    &iv, &aad, 128  // tag_bits
);
let mech = Mechanism::AesGcm(&gcm);

let plain = session.decrypt(
    &mech, aes_key, &ciphertext
)?;
// GCMタグ不一致 → Error::Pkcs11(CKR_ENCRYPTED_DATA_INVALID)
    AES-CBC との手順差

    CBCでは CKM_AES_CBC_PAD を使い、pParameter に16バイトのIVだけを渡す。タグはない。GCMより単純だが改ざん検知がない。

    CK_MECHANISM mech = { CKM_AES_CBC_PAD, iv_16bytes, 16 };

    ▸暗号操作の戻り値チートシート

    戻り値場面対処
    CKR_OK成功—
    CKR_ENCRYPTED_DATA_INVALIDGCMタグ不一致 / 改ざんデータを破棄
    CKR_BUFFER_TOO_SMALL出力バッファ不足pulEncryptedDataLen で確認後リトライ
    CKR_KEY_FUNCTION_NOT_PERMITTEDCKA_ENCRYPT=false の鍵適切な鍵を選ぶ
    CKR_MECHANISM_PARAM_INVALIDIV長など引数不正GCM: IV=12B, Tag=128bitを確認
    出典：[1] PKCS#11 Spec v3.2 §5.15[11] NIST SP 800-38D (GCM)

    CHAPTER 2 · 署名検証 · シーケンス

    06署名検証 — C_VerifyInit + C_Verify

    検証は公開鍵操作。秘密鍵へのアクセスは不要なので、ログイン不要で実行できる場合も多い。HSM内部では「ハッシュ計算 → 公開鍵でデパディング → ハッシュ比較」を行う。

    Verify の HSM処理は公開鍵演算なのでSign（秘密鍵）より速い。ログイン不要のケースも多い。
    Sign vs Verify の違い
    C_SignC_Verify
    使う鍵秘密鍵公開鍵
    ログイン必須多くは不要
    HSM演算重い（秘密鍵べき乗）軽い（公開鍵）
    戻り値署名バイト列CKR_OK / INVALID
    マルチパート検証
    C_VerifyUpdate(sess, chunk, len) // データを分割して渡す C_VerifyFinal(sess, sig, sigLen) // 最後に署名を渡して判定

    大きなファイルの検証で使う。ネットワーク型HSMではUpdateごとに往復が発生するため、ワンショットが推奨。

    出典：[1] PKCS#11 Spec v3.2 §5.16 Verify functions

    CHAPTER 2 · 署名検証 · VerifyRecover

    07C_VerifyRecover — メッセージ復元型署名

    C_VerifyRecover は RSA-PKCS（パディング方式）限定の特殊操作。通常の署名は「データを渡して、署名を返す」だが、RSAの性質上署名からデータを「復元」できる。ISO 9796-2などで使われる。

    C_VerifyRecoverInit + C_VerifyRecover
    C_VerifyRecoverInit(sess, &mech, hPubKey); C_VerifyRecover( sess, pSignature, ulSigLen, // 入力：署名 pData, &ulDataLen // 出力：復元された平文 );

    対応メカニズム：CKM_RSA_PKCS（PKCS#1 v1.5のみ）。PSSやECDSAは非対応。

    C_SignRecover との対称関係

    C_SignRecover は秘密鍵で「平文を署名に埋め込む」操作。C_VerifyRecover がその逆（公開鍵で取り出す）。

    操作入力出力
    C_SignRecover平文署名（平文埋め込み）
    C_VerifyRecover署名平文（復元）

    実際の使用例は少ない。TLS/PKI実務ではほぼ不要。

    出典：[1] PKCS#11 Spec v3.2 §5.16.5 C_VerifyRecover

    CHAPTER 3 · ダイジェスト

    08ダイジェスト — ハッシュと C_DigestKey

    PKCS#11のダイジェスト操作はデータのハッシュ計算をHSM内で行う。特殊な C_DigestKey を使うと鍵の素材をハッシュ入力に含めることができ、HMACの基礎操作に相当する。

    ▸基本フロー（3段ロケット）

    // SHA-256 ハッシュの取得 CK_MECHANISM mech = { CKM_SHA256, NULL_PTR, 0 }; C_DigestInit(sess, &mech); CK_BYTE hash[32]; CK_ULONG hLen = 32; C_Digest(sess, data, dataLen, hash, &hLen);

    ▸C_DigestKey — 鍵をダイジェストに混ぜる

    HMACとの違い

    PKCS#11には CKM_SHA256_HMAC という直接のHMACメカニズムもある（C_SignInitで使う）。C_DigestKey はより低レベルの操作。実用上はHMACメカニズムが推奨。

    ▸主なダイジェストメカニズム

    Mechanism出力長備考
    CKM_SHA_120 bytes非推奨（衝突脆弱性）
    CKM_SHA25632 bytes現代の標準。TLS/PKIで最も使われる
    CKM_SHA38448 bytesP-384/ECDSA-384とセット
    CKM_SHA51264 bytes高セキュリティ要件
    CKM_SHA3_25632 bytesSHA-3 (FIPS 202)。v2.40+
    出典：[1] PKCS#11 Spec v3.2 §5.17 Digest functions[12] FIPS 180-4

    CHAPTER 3 · ダイジェスト · 複合操作

    09複合操作 — 1回の往復で2つの操作を実行

    PKCS#11 v3.2 には「ダイジェスト + 暗号化」「署名 + 暗号化」等を一度のHSM通信で実行する複合関数がある。ネットワーク型HSMでは往復回数がそのままレイテンシになるため、有用。

    C_DigestInit + C_EncryptInit を両方完了済み。データを渡すと同時にハッシュ計算と暗号化を実行。 用途例：ファイルを暗号化しながらSHA-256を計算 → 別途 C_DigestFinal / C_EncryptFinal で完了 Digest + Encrypt C_DecryptDigestUpdate 前提：C_DecryptInit + C_DigestInit を両方完了済み。暗号文を渡すと同時に復号してハッシュ計算。 用途例：暗号ファイルを復号しながら内容のハッシュを計算 → 内容の完全性を確認 Decrypt + Digest C_SignEncryptUpdate 前提：C_SignInit + C_EncryptInit を両方完了済み。署名と暗号化を同時実行。 用途例：メッセージを署名しながら暗号化（Sign-then-Encrypt）→ C_SignFinal / C_EncryptFinal で完了 Sign + Encrypt C_DecryptVerifyUpdate 前提：C_DecryptInit + C_VerifyInit を両方完了済み。復号しながら署名を検証。 用途例：暗号化+署名されたメッセージを受け取り、復号と署名検証を同時に Decrypt + Verify
    使用パターン
    // DigestEncryptUpdate の使い方 C_DigestInit(sess, &sha256Mech); C_EncryptInit(sess, &aesMech, hKey); C_DigestEncryptUpdate( sess, pPart, ulPartLen, // 入力 pEncPart, &ulEncPartLen // 暗号化済み出力 ); C_DigestFinal(sess, hash, &hashLen); C_EncryptFinal(sess, final, &finalLen);
    ベンダー対応状況に注意

    複合操作は仕様に定義されているが、多くのHSMベンダーが未実装。SoftHSM2も一部未実装。

    C_GetMechanismList で確認するのはもちろん、実機で事前検証が必須。未対応の場合は個別に2回呼ぶ。

    出典：[1] PKCS#11 Spec v3.2 §5.17-5.18 Combined operations

    CHAPTER 4 · オブジェクト管理 · 検索

    10オブジェクト検索 — C_FindObjects 3段ロケット

    C_FindObjects は署名や暗号化と同じく3段構造。属性テンプレートで絞り込み → ハンドル一覧を取得。鍵のラベルや用途で検索し、ハンドルを取り出す日常的な操作。

    FindObjectsの落とし穴
    • ラベル検索は部分一致でない — 属性は完全一致。前後の空白も区別される。
    • Init/Final のペア忘れ — C_FindObjectsFinal を呼ばないと「検索アクティブ」状態が残り、別のC_FindObjectsInitが CKR_OPERATION_ACTIVE を返す。
    • maxCount を超えた場合 — 複数回 C_FindObjects を呼べば残りを取れる（内部カーソルが進む）。
    出典：[1] PKCS#11 Spec v3.2 §5.9 Object search functions

    CHAPTER 4 · オブジェクト管理 · 属性操作

    11属性の読み書き — Get / Set AttributeValue

    ハンドルを持っているオブジェクトから属性を読み出す、または書き換える。第1部で学んだ CKA_* 属性を実際に操作する関数。

    ▸C 言語コード例

    C
    /* --- 属性読み出し（2段階） --- */
CK_ATTRIBUTE tmpl[] = {
  { CKA_LABEL, NULL, 0 },  /* 1回目: サイズだけ */
};
pF->C_GetAttributeValue(sess, hKey, tmpl, 1);

/* tmpl[0].ulValueLen にラベル長が入る */
char *label = malloc(tmpl[0].ulValueLen);
tmpl[0].pValue = label;

pF->C_GetAttributeValue(sess, hKey, tmpl, 1);
/* label に値が入っている */

    ▸属性の変更可否まとめ

    属性読取変更備考
    CKA_LABEL○○自由に変更可
    CKA_ID○○識別子。変更可
    CKA_CLASS○×生成時に固定
    CKA_KEY_TYPE○×鍵種類は不変
    CKA_SENSITIVE○false→true のみ一方向
    CKA_EXTRACTABLE○true→false のみ一方向
    CKA_MODULUS○(公開鍵)×鍵素材は不変
    出典：[1] PKCS#11 Spec v3.2 §5.9 C_GetAttributeValue / §5.10 C_SetAttributeValue

    CHAPTER 4 · オブジェクト管理 · 生成・複製・削除

    12オブジェクトの生成・複製・削除

    C_GenerateKey/C_GenerateKeyPairとは別に、属性テンプレートから直接オブジェクトを作る C_CreateObject、属性を変えて複製する C_CopyObject、削除する C_DestroyObject。

    C_CreateObject の典型パターン
    C
    /* 証明書をTokenにインポート */
CK_OBJECT_CLASS cls = CKO_CERTIFICATE;
CK_CERTIFICATE_TYPE ctype = CKC_X_509;
CK_ATTRIBUTE tmpl[] = {
  { CKA_CLASS,            &cls,   sizeof(cls)   },
  { CKA_CERTIFICATE_TYPE, &ctype, sizeof(ctype) },
  { CKA_TOKEN,            &true_val, 1          },
  { CKA_LABEL,            "my-cert", 7          },
  { CKA_VALUE,            cert_der, cert_len     },
};
CK_OBJECT_HANDLE hCert;
pF->C_CreateObject(sess, tmpl, 5, &hCert);
    DestroyObject の注意点
    • R/Wセッションでログイン済みであること
    • CKA_DESTROYABLE=false の鍵は削除不可（v3.0+）
    • 削除前に C_CopyObject でバックアップすることを検討
    • HSMの監査ログには記録されるが、鍵の復元手段はない
    出典：[1] PKCS#11 Spec v3.2 §5.8 C_CreateObject / §5.8.3 C_DestroyObject

    CHAPTER 5 · 鍵エスクロー · WrapKey / UnwrapKey

    13鍵の安全な移送 — WrapKey / UnwrapKey

    Wrap = 鍵を別の鍵で暗号化して外に出す。平文で鍵が出ないので安全。HSM間の鍵移行、鍵バックアップ、鍵配布プロトコルで必須の操作。

    ラップで使う主なメカニズム
    Mechanism用途
    CKM_AES_KEY_WRAPRFC 3394 AES Key Wrap。鍵専用。
    CKM_AES_KEY_WRAP_PADパディング付き。任意長の鍵に対応。
    CKM_RSA_PKCS_OAEPRSA公開鍵で対称鍵を包む。鍵配布向き。
    CKM_AES_GCM汎用だがWrapに使えるかはベンダー依存。
    Wrap/Unwrap の前提条件
    • 対象鍵: CKA_EXTRACTABLE=true 必須
    • ラッピング鍵: CKA_WRAP=true
    • アンラッピング鍵: CKA_UNWRAP=true
    • R/Wセッション + ログイン済み
    出典：[1] PKCS#11 Spec v3.2 §5.15 Key wrapping[13] NIST SP 800-38F (AES Key Wrap)

    CHAPTER 5 · 鍵エスクロー · DeriveKey

    14鍵導出 — C_DeriveKey と ECDH 鍵合意

    C_DeriveKey は既存の鍵から新しい鍵を導出する操作。最も重要な用途はECDH鍵合意（Diffie-Hellman）。TLS 1.3 の鍵交換で中核となる操作。

    DeriveKey のその他の用途
    • CKM_CONCATENATE_BASE_AND_KEY — 2つの鍵素材を連結
    • CKM_XOR_BASE_AND_DATA — 鍵とデータのXOR
    • CKM_EXTRACT_KEY_FROM_KEY — 鍵の一部を切り出し
    • CKM_SP800_108_COUNTER_KDF — NIST SP 800-108 KDF (v3.0+)
    出典：[1] PKCS#11 Spec v3.2 §5.15 C_DeriveKey, §2.16 ECDH[14] NIST SP 800-56A (Key Agreement)

    CHAPTER 6 · 環境情報 · 基本情報取得

    15環境情報の取得 — ライブラリ・スロット・トークン

    HSMに接続した後、まず環境の基本情報を取得する3つの関数。ベンダー名、バージョン、利用可能なスロット数、トークンの容量などを確認する。

    tokenPresent=CK_TRUE にすると Token が挿入されたスロットだけに絞る。 2-call convention: pSlotList=NULL で count を取得 → バッファ確保 → 再呼出し。 CK_SLOT_ID slots[16]; CK_ULONG cnt = 16; C_GetSlotList (CK_TRUE, slots, &cnt); // slots[0..cnt]
    性能評価に使える情報

    CK_TOKEN_INFO の ulMaxSessionCount でHSMが同時に開けるセッション数の上限がわかる。これが並列ベンチマークの上限を決める。ulFreePublicMemory / ulFreePrivateMemory でToken内の残り容量も確認可能。

    出典：[1] PKCS#11 Spec v3.2 §5.5 Slot and token management

    CHAPTER 6 · 環境情報 · メカニズム照会

    16メカニズム照会 — 対応アルゴリズムの確認

    そのHSMが何に対応しているかを事前確認する関数。ベンダーごとの「方言」差をコードから検出する、性能評価の設計段階で必須の操作。

    ▸C_GetMechanismList — 一覧取得

    C
    /* 2-call convention */
CK_ULONG cnt = 0;
pF->C_GetMechanismList(slotId, NULL, &cnt);

CK_MECHANISM_TYPE *mechs = malloc(
    cnt * sizeof(CK_MECHANISM_TYPE));
pF->C_GetMechanismList(slotId, mechs, &cnt);

for (CK_ULONG i = 0; i < cnt; i++) {
    printf("  0x%08lx\n", mechs[i]);
}

    出力例: CKM_RSA_PKCS_KEY_PAIR_GEN, CKM_AES_GCM, CKM_SHA256, ...

    ▸C_GetMechanismInfo — 詳細照会

    C
    CK_MECHANISM_INFO info;
pF->C_GetMechanismInfo(
    slotId, CKM_RSA_PKCS_KEY_PAIR_GEN, &info);

printf("min key: %lu\n", info.ulMinKeySize);
printf("max key: %lu\n", info.ulMaxKeySize);
printf("flags: 0x%lx\n", info.flags);

    ▸CK_MECHANISM_INFO の中身

    フィールド意味
    ulMinKeySize最小鍵長（RSAなら 2048等）
    ulMaxKeySize最大鍵長（RSAなら 4096等）
    flagsCKF_SIGN / CKF_ENCRYPT / CKF_WRAP 等
    フラグの読み方（CK_MECHANISM_INFO.flags）
    フラグ値意味
    CKF_ENCRYPT0x00000100このMechanismで暗号化可
    CKF_DECRYPT0x00000200復号可
    CKF_SIGN0x00000800署名可
    CKF_VERIFY0x00002000検証可
    CKF_GENERATE_KEY_PAIR0x00010000鍵ペア生成可
    CKF_WRAP0x00020000鍵ラップ可
    CKF_DERIVE0x00080000鍵導出可
    出典：[1] PKCS#11 Spec v3.2 §5.5 C_GetMechanismList / C_GetMechanismInfo

    CHAPTER 6 · 環境情報 · セッション状態

    17セッション状態の照会と保存

    C_GetSessionInfo でセッションの現在状態を、C_GetOperationState / C_SetOperationState で進行中の操作をバイト列として保存・復元する。

    ▸C_GetSessionInfo

    C
    CK_SESSION_INFO info;
pF->C_GetSessionInfo(sess, &info);

printf("slot:  %lu\n", info.slotID);
printf("state: %lu\n", info.state);
printf("flags: 0x%lx\n", info.flags);
printf("error: %lu\n", info.ulDeviceError);

    ▸セッション状態の種類

    状態意味
    CKS_RO_PUBLIC_SESSION読取り専用 / 未ログイン
    CKS_RO_USER_FUNCTIONS読取り専用 / User ログイン済
    CKS_RW_PUBLIC_SESSION読書き可 / 未ログイン
    CKS_RW_USER_FUNCTIONS読書き可 / User ログイン済
    CKS_RW_SO_FUNCTIONS読書き可 / SO ログイン済

    ▸C_GetOperationState / C_SetOperationState

    マルチパート操作（DigestUpdate中など）の途中状態をバイト列としてエクスポートし、後で復元できる。

    ベンダー実装差

    C_GetOperationState が CKR_STATE_UNSAVEABLE を返す場合は「この操作の途中保存は非対応」。特に署名の途中状態は多くのHSMで非対応。

    出典：[1] PKCS#11 Spec v3.2 §5.6 C_GetSessionInfo, §5.7 C_GetOperationState
