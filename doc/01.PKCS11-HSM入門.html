    PART I — 概念

    03HSMとは何か

    HSM（Hardware Security Module）は、鍵を保管し、暗号処理を「中で」実行する専用ハードウェア。アプリには鍵を渡さず、処理だけを請け負う。

    何をしてくれる？
    • 鍵の生成・保管
    • 署名・検証
    • 暗号化・復号
    • 乱数生成
    • 鍵のラップ（暗号化して持ち出し）
    絶対にしないこと
    • 秘密鍵を外に出すこと
      （属性で許可した場合の例外を除く）
    • アプリのメモリに鍵を載せること
    • 認証なしで秘密操作をすること

    PART I — 概念

    04PKCS#11とは何か

    暗号デバイス（HSM、スマートカード等）を操作するためのC言語API標準仕様。OASIS PKCS 11 TCが管理。別名 Cryptoki。最新は v3.2（2024）。

    ▸たとえると「統一注文書フォーマット」

    HSMメーカーは Thales、Utimaco、AWS、Yubico など多数あって中身はバラバラ。でも 外向けの注文書だけは同じ書式で受け付けようや という業界合意。

    SuicaとICOCAとPASMOの関係に似てる。中身は違うけど、レジでの読み取り操作は同じ。

    アプリ開発者にとっての意味

    ベンダー提供の .so（Linux）や .dll（Windows）を ロードして、決まった関数を呼ぶ。HSMメーカーを変えてもアプリ側のロジックは（基本的に）動く。※ 実際はベンダーごとの方言あり

    出典：[1] PKCS#11 Spec v3.2[5] OASIS PKCS 11 TC

    05全体アーキテクチャ

    ネットワーク経由でアクセス。

    PART II — 5つの基本概念

    06Slot と Token

    Slot = 棚の番号、Token = その棚に置いてある実際の箱。1台のHSMを複数チームで分割使用するときの単位にもなる。

    Slot（スロット）

    棚の番号。実機HSMでは「パーティション」と呼ばれることが多い。物がなくても番号は存在する。

    関数：C_GetSlotList / C_GetSlotInfo

    Token（トークン）

    その棚に置いてある実際の箱。鍵やデータが入る器。ラベル名、製造元、対応Mechanism等のメタ情報を持つ。

    関数：C_GetTokenInfo

    出典：[1] PKCS#11 Spec v3.2 §5
    07 / 19
    PART II — 5つの基本概念

    07Session

    セッション = アプリとTokenの間の論理接続。電話と同じ感覚で、開いて、用件を済ませて、閉じる。

    ⚠マルチスレッドの大原則

    1セッションを複数スレッドで共有しない。途中で別スレッドの操作が入ると、セッションの「今、何の操作中か」がぐちゃぐちゃになる。

    スレッドごとに別のC_OpenSessionを持つのがルール。

    出典：[1] PKCS#11 Spec v3.2 §5.6 Session
    08 / 19
    PART II — 5つの基本概念 (★ 最重要)

    08Object と Handle

    HSMの本質はここ。鍵そのものはアプリに出てこない。代わりに整理番号（Handle）を受け取って、「47番でお願いします」と頼む。

    ▸普通のソフトウェア暗号

    key = "ABCDEF123456..."
       ↑ 鍵そのものがメモリに乗る
sign(key, data)
       ↑ 流出リスク

    ▸HSM経由

    key_handle = 47
       ↑ ただの整数（預かり証）
sign(key_handle, data)
       ↑ 鍵はHSM内で完結
    なぜハンドル方式？

    PCがマルウェアに感染しても、メモリには整数しかないから鍵自体は盗まれない。これがHSMを使う最大の理由。

    PART II — 5つの基本概念

    09Attribute（属性）

    Object（鍵・証明書・データ）はすべて属性（タグ）の集合として表現される。鍵を作るときも検索するときも、属性で指定する。

    ▸主な属性の意味

    CKA_CLASS
    オブジェクトの種類（秘密鍵・公開鍵・証明書等）
    CKA_KEY_TYPE
    鍵のアルゴリズム（RSA / EC / AES）
    CKA_LABEL
    ユーザー定義の名前。検索に使う
    CKA_SIGN / CKA_VERIFY
    用途フラグ（署名OK・検証OK等）
    CKA_SENSITIVE
    秘密値の保護フラグ
    CKA_EXTRACTABLE
    外部に持ち出し可能か（要注意）
    CKA_TOKEN
    永続オブジェクト（true）／揮発（false）
    設計上のキーポイント

    CKA_EXTRACTABLE を true にすると、鍵を暗号化して外に出せる。用途次第やけど、本番運用では基本 false。一度falseにしたら原則戻せない（ベンダー差あり）。

    出典：[1] PKCS#11 Spec v3.2 §10 Object types[8] AWS CloudHSM Key attributes
    10 / 19
    PART II — 5つの基本概念

    10Mechanism（メカニズム）

    暗号操作の「種類」「方式」を指定する仕様。鍵とメカニズムは別物。料理でいうと「材料」と「レシピ」の関係。

    ▸同じRSA鍵でも色んな使い方

    CKM_RSA_PKCS
    古いパディング方式（PKCS#1 v1.5）
    CKM_RSA_PKCS_PSS
    PSSパディング（現代の推奨）
    CKM_RSA_PKCS_OAEP
    暗号化用パディング
    CKM_SHA256_RSA_PKCS
    ハッシュ込みのコンビ品

    ▸カテゴリ別の代表

    RSA系
    CKM_RSA_PKCS_PSS, CKM_RSA_PKCS_OAEP
    EC系
    CKM_ECDSA_SHA256, CKM_ECDH1_DERIVE
    AES系
    CKM_AES_CBC_PAD, CKM_AES_GCM
    ハッシュ
    CKM_SHA256, CKM_SHA512
    鍵生成
    CKM_RSA_PKCS_KEY_PAIR_GEN
    注文書のイメージ

    「鍵 = #47」「メカニズム = RSA-PSS」「データ = ファイルの中身」
    この3点セットを毎回 C_SignInit + C_Sign で渡す。

    ⚠ベンダー差の罠

    仕様書に載ってるMechanismでも、HSMが対応してない場合がある。最新のEdDSAとか、PSSの細かいパラメータとか。事前に C_GetMechanismList で対応一覧を取って確認する。

    出典：[1] PKCS#11 Spec v3.2 §6 Mechanisms[7] CloudHSM Mechanisms

    PART III — ライブラリの内側

    11ライブラリ内部の4つの役割

    「窓口係」と一言で呼んできたけど、実際は4つの仕事を兼ねてる。性能を理解するにはこの分担を意識する。

    ① 受付係 / ② 帳簿係

    標準APIの窓口と内部状態の管理。マルチスレッド性能はここの設計次第でスケールが決まる。

    ③ 翻訳係 / ④ 配達係

    ネットワーク型HSMでは④の通信レイテンシが性能の大部分を占めることが多い。TLS接続は使い回しが必須。

    PART III — ライブラリの内側

    12実装パターン4種

    同じPKCS#11ライブラリでも、HSMとの通信手段によって性能特性がガラッと変わる。

    A — ソフトウェアHSM

    例：SoftHSM2、OpenSC

    アプリ → .so → ローカルファイル

    用途: 開発・学習・API動作確認

    性能評価には使えない（CPUだけ測る形になる）

    B — ネットワーク型HSM (★今回の対象)

    例：AWS CloudHSM、Thales Luna

    アプリ → .so → TLS → アプライアンス

    特徴: 1リクエストごとにネットワーク往復

    レイテンシの大半がネットワーク。コネクション使い回し必須。

    C — PCIeカード型

    例：Thales ProtectServer、Marvell LiquidSecurity

    アプリ → .so → ドライバ → PCIeバス

    特徴: 超低レイテンシ

    1台のサーバに数枚しか挿せない

    D — USBトークン型

    例：YubiHSM、スマートカード

    アプリ → .so → libusb / PC/SC

    特徴: 個人/小規模向け、低コスト

    USBプロトコルのレイテンシが乗る

    性能ボトルネックの分布
    パターンボトルネック備考
    A (ソフト)CPU・暗号演算ライブラリ自体のオーバーヘッド測定に有用
    B (ネット)ネットワーク往復RTTが支配的、並列化で稼ぐ
    C (PCIe)HSM内部演算ほぼピュアな暗号演算性能が出る
    D (USB)USBプロトコルシリアル通信に近い特性
    出典：[6] SoftHSM2[7] AWS CloudHSM[10] Thales Luna[11] NIST CMVP

    PART IV — 関数と処理フロー

    13関数の命名規則

    PKCS#11の関数は すべて C_ で始まる。名前自体が役割を語るように設計されてる。

    ▸動作の例

    • Open / Close — 開く / 閉じる
    • Get / Set — 取得 / 設定
    • Find — 検索
    • Generate — 生成
    • Sign / Verify — 署名 / 検証
    • Encrypt / Decrypt — 暗号化 / 復号
    • Wrap / Unwrap — 鍵のラップ

    ▸対象の例

    • Session — セッション
    • SlotList / SlotInfo — スロット
    • TokenInfo — トークン情報
    • Object / Objects — オブジェクト
    • Key / KeyPair — 鍵
    • AttributeValue — 属性値
    • Random — 乱数

    英語が読めれば関数名から動作を推測できる。これがPKCS#11の親切なところ。一度パターンを覚えたら、ベンダーが変わっても同じ関数名で書ける。

    PART IV — 関数と処理フロー

    14機能カテゴリマップ

    PKCS#11の全関数は大きく7グループに分類できる。性能評価で叩くのは主に⑥。

    カテゴリ代表的な関数役割
    ① 全体管理 C_Initialize / C_Finalize / C_GetFunctionList ライブラリの起動と終了
    ② Slot/Token情報 C_GetSlotList / C_GetTokenInfo / C_GetMechanismList 環境情報の取得
    ③ セッション C_OpenSession / C_CloseSession / C_Login / C_Logout 接続と認証
    ④ オブジェクト C_FindObjects... / C_GetAttributeValue / C_CreateObject 鍵や証明書の検索・属性操作
    ⑤ 鍵管理 C_GenerateKey / C_GenerateKeyPair / C_WrapKey / C_UnwrapKey 鍵の生成・ラップ
    ⑥ 暗号操作 C_Sign... / C_Verify... / C_Encrypt... / C_Decrypt... / C_Digest... / C_GenerateRandom ★ 性能評価のメイン対象
    ⑦ その他 C_SeedRandom / C_WaitForSlotEvent 付加機能
    性能評価の設計指針

    ①〜⑤は準備フェーズ。⑥を繰り返し叩く部分が計測フェーズ。これをはっきり分けておくと、ベンチマーク結果の解釈が楽になる。

    出典：[1] PKCS#11 Spec v3.2 §5 Function categories

    PART IV — 関数と処理フロー

    153段ロケット構造

    PKCS#11の最大のクセ。データを処理する操作はだいたい3段階に分かれる。

    ▸操作別のセット

    操作Init（準備）本処理Final（分割時）
    署名C_SignInitC_SignC_SignFinal
    検証C_VerifyInitC_VerifyC_VerifyFinal
    暗号化C_EncryptInitC_EncryptC_EncryptFinal
    復号C_DecryptInitC_DecryptC_DecryptFinal
    ハッシュC_DigestInitC_DigestC_DigestFinal
    検索C_FindObjectsInitC_FindObjectsC_FindObjectsFinal
    なぜ3段？

    セッションが「今、何の操作中か」を覚えてる必要があるから。C_Sign単独では呼べない。先にC_SignInitで「これからどの鍵で、どのMechanismで」を宣言しておく。

    単発で完結する操作

    C_GenerateKey、C_GenerateRandom、C_GetAttributeValue、C_WrapKey などは単発。「データを処理する系」だけ3段と覚える。

    出典：[1] PKCS#11 Spec v3.2 §5.13-5.18

    PART IV — 関数と処理フロー

    16完全ライフサイクル

    起動から1回署名して終わるまでの全14ステップ。銀行のたとえも併記。

    00
    .so ファイルをロード
    動的ロード機能で libcloudhsm.so 等を読み込む
    ↓
    01
    C_GetFunctionList
    関数メニュー表を取得。以降の呼び出しはこのメニュー経由
    🏦 「使えるサービス一覧ください」
    ↓
    02
    C_Initialize
    ライブラリ起動。CKF_OS_LOCKING_OKでマルチスレッド有効化
    🏦 「営業開始です」
    ↓
    03
    C_GetSlotList
    利用可能なスロット番号の配列を取得
    🏦 「使える棚どれ？」
    ↓
    04
    C_OpenSession
    セッションハンドルを取得（マルチスレッド時はスレッドごと）
    🏦 「整理券ください」
    ↓
    05
    C_Login
    User PIN認証。これ以降Private属性に触れる
    🏦 「暗証番号は1234です」
    ↓
    06-08
    C_FindObjectsInit → C_FindObjects → C_FindObjectsFinal
    属性テンプレートで鍵を検索。鍵ハンドルが返ってくる
    🏦 「『印鑑』ってラベルのやつ探して」
    ↓
    09
    C_SignInit
    「鍵#47でRSA-PSS署名」と宣言。セッションが署名待ち状態に
    🏦 「47番の印鑑で判子押す準備して」
    ↓
    ★10
    C_Sign
    データを渡して署名取得。性能評価のメイン計測点
    🏦 「この書類に押してください」
    ↓
    11-12
    C_Logout → C_CloseSession
    ログアウトとセッション終了
    🏦 「整理券返します」
    ↓
    13
    C_Finalize
    ライブラリ終了
    🏦 「営業終了」
    出典：[1] PKCS#11 Spec v3.2 §5[9] CloudHSM PKCS#11 Examples

    PART V — 性能評価

    17性能評価の構造

    毎回フル14ステップやってたら準備時間ばかり測ることに。準備と片付けは1回、本処理だけループが原則。

    準備
    測定対象外
    .soロード → C_GetFunctionList → C_Initialize → C_GetSlotList → C_OpenSession → C_Login → C_FindObjects...
    ↓
    ウォーム
    アップ
    測定対象外（初回キャッシュミス対策）
    for i in 1..100: C_SignInit + C_Sign（空打ち）
    ↓
    ★計測
    この区間だけ測る
    for i in 1..N: t0 = now() → C_SignInit + C_Sign → t1 = now() → latency[i] = t1 - t0
    ↓
    集計
    統計量を計算
    平均、p50、p95、p99、max、min、スループット（ops/sec）
    ↓
    片付け
    C_Logout → C_CloseSession → C_Finalize
    よくある測定の歪み
    • ウォームアップ忘れで初回TLS確立が混入
    • 戻り値チェック忘れでエラー結果も計測
    • C_Signを呼ぶ前にC_SignInitを忘れる
    • 鍵生成と署名を同じ枠で測る（外れ値が出る）
    エラーハンドリング

    すべての関数はCK_RV型を返す。CKR_OK = 0 なら成功、それ以外はエラー。

    よく見る：CKR_SESSION_HANDLE_INVALID, CKR_USER_NOT_LOGGED_IN, CKR_MECHANISM_INVALID

    出典：[1] PKCS#11 Spec v3.2 §5.1 Return values[3] PKCS#11 Usage Guide v3.2
